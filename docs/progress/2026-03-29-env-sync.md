# March 29, 2026: Milestone 7 — Environment Variable Sync

## What was done

### `devassemble env pull`

Pulls all environment variables from the Vercel project (from the latest launch run)
and writes them to `.env.local` in the current directory.

- Finds the Vercel project ID from the most recent completed run in the state store
- Calls Vercel API `GET /v9/projects/{id}/env?decrypt=true` to read decrypted values
- Writes sorted key=value pairs to `.env.local` with a header comment
- Shows each variable name in the output

**Use case**: New team member joins, runs `devassemble env pull`, gets all secrets
locally in 5 seconds instead of asking around for 2 hours.

### `devassemble env push`

Pushes local environment variables from `.env.local` (or `.env`) to the Vercel project.

- Reads `.env.local` first, falls back to `.env`
- Parses the file (skips comments and blank lines)
- Pushes each variable to Vercel with `upsert=true` (creates or updates)
- Targets both preview and production environments

**Use case**: Developer adds a new API key locally, runs `devassemble env push` to
sync it to Vercel without touching the dashboard.

### Design decisions

**No dotenv dependency**: Used a simple inline parser instead. `.env` files are a
trivial format (key=value, skip comments/blanks). Adding a dependency for this would
be overkill.

**`.env.local` as default output**: This is the Next.js convention for local-only
env files that are gitignored. Avoids accidentally committing secrets.

**Decrypt=true on pull**: The Vercel API returns encrypted values by default. We pass
`decrypt=true` to get the actual values. This requires the token to have read access.

**Upsert on push**: Uses `upsert=true` so pushing is idempotent — safe to run multiple
times.
